1. Süsteemi kontekst

Süsteem koosneb kuuest Docker Compose teenusest. Frontend SPA suhtleb backend REST API-ga. Backend orkestreerib äriloogikat, RBAC-d, auditlogi ja rentniku isoleerimist, kasutades PostgreSQL-i, Redis-it ja MinIO-d.

2. Backend arhitektuur

Python 3.11+ FastAPI rakendus SQLAlchemy 2.x ORM-i, Alembic migratsioonide, JWT autentimise ja RBAC-ga. Koos eksisteerib kaks API versiooni: v1 (pärand, 15 marsruutijat) ja v2 (aktiivne, 24 marsruutijat Tier A/B mitme rentniku arhitektuuriga). V2 on ainus rakenduse sisenemispunkti ühendatud versioon.

2.5 Migratsioonid

50 Alembic migratsioonifaili kataloogis backend/alembic/versions/ jälgivad skeemi täielikku arengut alates algsetest tabelitest (2024) läbi v2 E-ITS Tier A/B ümberkorralduse, IMR hetktõmmiste, turbeviiside ja mitmete liitmigratsioonide. Kõik migratsioonid on tagasipööratavad.

3. Frontend arhitektuur

React 18 + TypeScript SPA, ehitatud Vite 5-ga, kujundatud Tailwind CSS 3 + shadcn/ui (Radix aluskomponendid). Autentimise olekut haldab kohandatud use-auth hooks, JWT salvestatakse localStorage-s. API päringud lähevad läbi jagatud Axios eksemplari, mis lisab päistesse Authorization: Bearer ja X-Tenant-ID.

5. Turvamudel

AspektRakendus
AutentimineJWT (HS256) python-jose kaudu. Bearer token Authorization päises.
Parooliräsindusbcrypt passlib kaudu.
MFAToetatud mudelis (global_users.mfa_secret), realiseeritud pyotp kaudu.
AutoriseerimineRBAC rolli-õiguse vastendusega. Adminil on kõik õigused.
Rentniku isoleerimineIgal rentnikupõhisel tabelil on tenant_id. Päringud filtreeritakse X-Tenant-ID päise järgi.
AuditlogiMuudetamatu audit_logs tabel enne/pärast JSON hetktõmmistega.
FailihoidlaMinIO (S3-ühilduv). Tõendid on vaikimisi privaatsed.
Failide üleslaadimise kontrollTüübi- ja suuruskontroll API kihil.
CORSSeadistatav BACKEND_CORS_ORIGINS keskkonnamuutujaga.
TurvapäisedSeadistatud ainult Vite arendusserveris (CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy).
Turvalünk
Tootmise nginx konfiguratsioon (infra/nginx/nginx.conf) ei sea Content-Security-Policy ega muid turvapäiseid. CSP on konfigureeritud ainult Vite arendusserveris (frontend/vite.config.ts:20-30) ja sisaldab 'unsafe-inline' ja 'unsafe-eval', mis on vajalikud HMR-i jaoks arenduses, kuid tuleks tootmises eemaldada.

6.2 Autentimise voog

1. POST /api/v2/auth/login  (e-post + parool)
2. Server kontrollib mandaate, tagastab JWT + kasutaja info
3. Frontend salvestab tokeni localStorage-sse
4. Iga järgnev päring sisaldab:
     Authorization: Bearer <token>
     X-Tenant-ID: <org_uuid>
5. Server dekodeerib JWT, otsib kasutaja, kontrollib liikmelisust
6. Marsruudi käsitleja võib kutsuda require_role() või has_permission()

6.3 IMR genereerimise voog

1. Kasutaja vastendab E-ITS moodulid sihtmärgiga (vara või BP)
2. POST /api/v2/imr/generate
3. Teenus lahendab mooduli → meetmed eits_module_measures kaudu
4. Loob ImrItem kirjed (üks meetme kohta)
5. Hüppab üle duplikaatidest (sama rentnik + sihtmärk + versioon + meede)
6. Tagastab loodud IMR kirjed
7. Kasutaja saab määrata omanikud, prioriteedid, tähtajad

6.4 Tõendi üleslaadimise voog

1. Kasutaja laadib faili POST /api/v2/evidence/upload
2. Backend kontrollib faili tüüpi ja suurust
3. Fail salvestatakse MinIO panges: eits-evidence/
4. Tõendi metaandmed salvestatakse PostgreSQL-i
5. Tõendit saab siduda mis tahes üksusega evidence_links kaudu
   (target_type: imr_item | measure | risk | asset | bp)
