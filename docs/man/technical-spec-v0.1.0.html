1. System Context

The system runs as six Docker Compose services. The frontend SPA communicates with the backend REST API. The backend orchestrates business logic, RBAC, audit logging, and tenant isolation, backed by PostgreSQL, Redis, and MinIO.

2. Backend Architecture

Python 3.11+ FastAPI application with SQLAlchemy 2.x ORM, Alembic migrations, JWT authentication, and RBAC. Two API versions co-exist: v1 (legacy, 15 routers) and v2 (active, 24 routers with Tier A/B multi-tenancy). Only v2 is mounted on the application entrypoint.

2.5 Migrations

50 Alembic migration files in backend/alembic/versions/ track the full schema evolution from initial tables (2024) through the v2 E-ITS Tier A/B restructuring, IMR snapshots, protection mode, and various merge heads. All migrations are reversible where practical.

3. Frontend Architecture

React 18 + TypeScript SPA, built with Vite 5, styled with Tailwind CSS 3 + shadcn/ui (Radix primitives). Auth state managed via custom use-auth hook with JWT stored in localStorage. API calls go through a shared Axios instance that injects Authorization: Bearer and X-Tenant-ID headers.

5. Security Model

AspectImplementation
AuthenticationJWT (HS256) via python-jose. Bearer token in Authorization header.
Password hashingbcrypt via passlib. Hash rounds: default.
MFASupported in model (global_users.mfa_secret), implemented via pyotp.
AuthorizationRBAC with role-permission mapping. Admin has all permissions implicitly.
Tenant isolationEvery tenant-scoped table has tenant_id. Queries filter by X-Tenant-ID header.
Audit loggingAppend-only audit_logs table with before/after JSON snapshots.
File storageMinIO (S3-compatible). Evidence files are private by default.
File upload validationType and size checks at API layer.
CORSConfigured via BACKEND_CORS_ORIGINS env var.
Security headersSet on Vite dev server only (CSP, X-Content-Type-Options, X-Frame-Options, Referrer-Policy).
Security Gap
The production nginx config (infra/nginx/nginx.conf) does not set Content-Security-Policy or any other security headers. The CSP is only configured on the Vite dev server (frontend/vite.config.ts:20-30) and includes 'unsafe-inline' and 'unsafe-eval', which are required for HMR in development but should be removed in production.

6.2 Authentication Flow

1. POST /api/v2/auth/login  (email + password)
2. Server validates credentials, returns JWT + user info
3. Frontend stores token in localStorage
4. Every subsequent request includes:
     Authorization: Bearer <token>
     X-Tenant-ID: <org_uuid>
5. Server decodes JWT, looks up user, checks membership
6. Route handler may call require_role() or has_permission()

6.3 IMR Generation Flow

1. User maps E-ITS modules to a target (asset or BP)
2. POST /api/v2/imr/generate
3. Service resolves module → measures via eits_module_measures
4. Creates ImrItem records (one per measure)
5. Skips duplicates (same tenant + target + version + measure)
6. Returns generated IMR items
7. User can then assign owners, priorities, due dates

6.4 Evidence Upload Flow

1. User uploads file via POST /api/v2/evidence/upload
2. Backend validates file type and size
3. File stored in MinIO bucket: eits-evidence/
4. Evidence metadata stored in PostgreSQL
5. Evidence can be linked to any entity via evidence_links
   (target_type: imr_item | measure | risk | asset | bp)
